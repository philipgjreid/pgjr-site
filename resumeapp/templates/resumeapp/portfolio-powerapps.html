Power Apps Risk Tool Project

Project Overview

While I was working at the Ministry of Justice (circa 2024) as a risk consultant, I developed a proof-of-concept app for risk and assurance management using Power Apps. The existing risk tool had been created to move away from using Excel based risk registers, but still had several limitations and missing functionality. My aim was to show what could be done using Power Apps as a potential future tooling option.

What I Did

  • Current State Assessment - Assessed the current state and documented future state requirements as a baseline for creating the PoC app
  • Developed PoC Risk Tool Developed an app to showcase the potential functionality that could be built using Power Apps
  • Business Case for Risk Tool - Presented the PoC app to my team and incorporated the option into a draft business case

Skills Used

  • Power Apps - App development with DAX and Power Fx
  • Databases - Database structure creation in Dataverse

Project Details

Background

The existing solution had previously moved from using Excel based risk registers to creating a risk tool in Oracle Application Express. While this tool provided a better central repository for managing risks, the solution had several major limitations. This was exacerbated by the extension of assurance processes that required additional details to be managed.

Limitations of Existing Tool

  • Tracking of Control and Treatment Details - The tool was relatively simple with a single record for each risk. It tracked risk controls and treatments using free-text fields for each risk. This made tracking of updates over time was highly dependent on consistent data entry and any trend reporting was unsupportable.
  • User Access Management - The tool had three levels of access. View access allowed a user to see all the risks. Edit allowed a user to view and edit only risks where they were assigned as an owner. Admin allowed a user full access, which was meant for those supporting the tool only. The lack of flexibility in the View and Edit access options meant my team was left to update the tool to avoid confusion for staff.

Processes to Support

  • Assurance Process Extension - To support more consistent risk management, the assurance process was being matured to cover key risk controls across the group. This required risk controls to be better defined and the ability to track control assessments.
  • Risk Process Maturity - The existing tool was intended to support access by staff from across the group, but this had not happened due to the user access limitations. This made the risk process more difficult to embed with staff and more manual to support by my team.

Current State Assessment

To outline what functionality to include in the PoC I documented the current state and future requirements to help determine what to include. This was all part of a wider piece of work to consider what risk tooling options were available so they could be considered at a suitable time.

Current State

  • Current State Functionality - Documented the existing functionality so that if anyone else picked it up there would be a clear understanding of what had been reviewed.
  • Future Requirements for PoC - Outlined the main requirements, which included:
    • Control Library - Having a standalone register (or Control Library) for all the key risk controls was a key requirement to improve the current state and provide functionality similar to that found in an enterprise risk tool.
    • Treatment Register - Similar to the Control Library, have a separate register to track treatment activities would better support operational processes and be more like an enterprise risk tool.

Business Case Options

  • Range of Options - To consider future tooling options for risk and assurance processes there were 3 levels to be considered:
    • Enterprise Solution - An enterprise grade solution that would be sourced from an external supplier.
    • In-house Solution - A solution built within the organisation with the support of the ICT group. This was where PowerApps was seen as being the logical choice for replacing existing Oracle Application Express based tools.
    • Team-based Solution - Continue to improve processes by leveraging fully accessible Microsoft Office products like Excel and SharePoint to supplement the current tool.
  • Limitations on Power Apps - The ICT group was not completely ready to support the use of Power Apps, so drafting a business case was a way to store the findings of the PoC until a suitable time for consideration.

Developed PoC Risk Tool

Developing the PoC required sketching out an overall view of the tables and workflows required to create that app. Then it was a matter of creating a simple layout that could technically meet the main requirements.

Overall Structure

The data tables and screens (user interfaces) the app was going to need to collect and store the relevant data and allow users to view and edit the data as required. The main tables were:

  • Risk Register - Table for all of the risks.
  • Control Library - Table for all of the controls.
  • Risk to Control Link Table - A table to record the links between risks and controls. Note that this could be done automatically due to the Dataverse creating this type of table for many-to-many relationships, but I did this manually to have more control over the PoC approach.
  • Control Assessments - A table to store all control assessment details.
  • Treatment Register - A table to store all treatment details.

Workflows

The screens and workflows to allow users to interact with and update details as required. The main workflows to manage were:

  • Viewing Details - How to provide users with the ability to view what they need to. This included functionality like searching and filtering tables.
  • Editing Details - How to allow users to create or modify data. This ranged from simple create or modify tasks to more complex updates to one-to-many relationships.
  • Deleting Details - How to allow users to delete elements with suitable protections in place, such as pop-up boxes to confirm the action being requested.

Technical Challenges

  • Linking Risks to Controls - This was the main requirement to fulfil. Having a many-to-many relationship between risks and controls was key to be able to create a tool that worked more like an enterprise tool. Creating a user interface where users could add and remove these links was achieved and showcased how this type of functionality makes the risk process much easier to engage with.
  • Updating Details - The basic approach in Power Apps is to use a Form (an input form that aligns directly to a table). This works well, but is inflexible. Using Patch commands is more flexible, so these were used in a couple of places to allow update interfaces to be more user-friendly.
  • App Formatting - While the formatting of the PoC was not a priority, using the built-in formatting still required some effort to get the PoC into a structure where it was presentable.

Final Result

I presented the PoC tool to my team, which was well received and helped give them a sense of what might be possible using Power Apps. The draft business case was updated for future reference.
