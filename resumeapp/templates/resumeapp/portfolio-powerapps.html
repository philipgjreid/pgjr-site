Power Apps Risk Tool Project

Project Overview

While working at the Ministry of Justice in 2024 as a senior risk consultant, I developed a proof-of-concept (PoC) app for risk and assurance management using Power Apps. The existing risk tool had moved away from Excel-based risk registers but still faced several limitations and lacked essential functionality. My aim was to demonstrate the potential capabilities of Power Apps as a future tooling option.

What I Did

  • Current State Assessment - Assessed the current state and documented future state requirements as a baseline for creating the PoC app
  • Developed PoC Risk Tool - Created an app to showcase the potential functionality that could be built using Power Apps
  • Business Case for Risk Tool - Presented the PoC app to my team and incorporated the option into a draft business case

Skills Used

  • Power Apps - App development with DAX and Power Fx
  • Databases - Database structure creation in Dataverse

Project Details

Background

The existing solution had previously transitioned from Excel-based risk registers to a risk tool in Oracle Application Express. Although this tool provided a better central repository for managing risks, it had several major limitations. These issues were exacerbated by the extension of assurance processes, which required additional details to be managed.

Limitations of Existing Tool

  • Tracking of Control and Treatment Details - The tool was relatively simple, with a single record for each risk. It tracked risk controls and treatments using free-text fields, making tracking updates over time highly dependent on consistent data entry. Consequently, any trend reporting was unsupportable.
  • User Access Management - The tool had three levels of access. View access allowed a user to see all risks. Edit access allowed a user to view and edit only risks where they were assigned as an owner. Admin access allowed full access, intended for those supporting the tool. The lack of flexibility in the View and Edit access options meant my team had to update the tool to avoid confusion for staff.

Processes to Support

  • Assurance Process Extension - To support more consistent risk management, the assurance process was matured to cover key risk controls across the group. This required better-defined risk controls and the ability to track control assessments.
  • Risk Process Maturity - The existing tool was intended to support access by staff from across the group, but this had not happened due to user access limitations. This made the risk process more difficult to embed with staff and more manual for my team to support.

Current State Assessment

To outline what functionality to include in the PoC, I documented the current state and future requirements to help determine what to include. This was part of a broader effort to consider available risk tooling options for future implementation.

Current State

  • Current State Functionality - Documented existing functionality to ensure clear understanding if anyone else took over the project.
  • Future Requirements for PoC - Outlined the main requirements, which included:
    • Control Library - A standalone register (or Control Library) for all key risk controls, providing functionality similar to an enterprise risk tool.
    • Treatment Register - A separate register to track treatment activities, supporting operational processes and resembling an enterprise risk tool.

Business Case Options

  • Range of Options - To consider future tooling options for risk and assurance processes, three levels were explored:
    • Enterprise Solution - An enterprise-grade solution from an external supplier.
    • In-house Solution - A solution built within the organization with ICT group support, where PowerApps was the logical choice to replace existing Oracle Application Express tools.
    • Team-based Solution - Improve processes using Microsoft Office products like Excel and SharePoint to supplement the current tool.

Developed PoC Risk Tool

Developing the PoC required sketching out an overall view of the tables and workflows needed to create the app. The goal was to create a simple layout that could technically meet the main requirements.

Overall Structure

The data tables and screens (user interfaces) the app was going to need to collect and store the relevant data and allow users to view and edit the data as required. The main tables were:

  • Risk Register - Table for all risks.
  • Control Library - Table for all controls.
  • Risk to Control Link Table - Table to record links between risks and controls.
  • Control Assessments - Table to store all control assessment details.
  • Treatment Register - Table to store all treatment details.

Workflows

The screens and workflows to allow users to interact with and update details as required. The main workflows to manage were:

  • Viewing Details - Allow users to view necessary information, including search and filter functionality.
  • Editing Details - Enable users to create or modify data, from simple tasks to complex updates to one-to-many relationships.
  • Deleting Details - Allow users to delete elements with appropriate protections, such as confirmation pop-ups.

Technical Challenges

  • Linking Risks to Controls - Achieving a many-to-many relationship between risks and controls to create a more enterprise-like tool.
  • Updating Details - Using flexible Patch commands instead of basic Forms to create more user-friendly update interfaces.
  • App Formatting - Ensuring the PoC was presentable using built-in formatting, despite formatting not being the primary focus.

Final Result

I presented the PoC tool to my team, which was well received. It demonstrated the potential of using Power Apps for risk and assurance management. The draft business case was updated to reflect the findings and for future reference.
