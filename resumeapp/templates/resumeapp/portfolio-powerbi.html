Project Details

Data Analysis and Requirement Setting

The normal first step for creating any dashboard or performance reporting content is to start with user requirements. In this case I did not have any users, so I considered the available data to determine the requirements.

Reviewed Existing Data

The data from the ergast dataset extends back to 1950 and has details on races, drivers, and circuits for each F1 season.

  • Amount of Data - The dataset was large and provided a lot of data to consider for inclusion from both the time period covered and the amount of detail available for each race, which extends down to lap times for each driver. Some data points were only available from more recent years, so this also had to be taken into account.
  • Potential Data Views - The dataset provided several different ways to view the data, such as from a specific driver's perspective. This meant a range of views and levels of detail could be considered.

Requirements Gathering

The requirements were based around what someone interested in F1 might like to know. The main requirements are:

  • Overview of F1 History - Provide access to an overview of F1 historical results.
  • Easy Navigation to Points of Interest - Allow users to easily navigate into subject areas of interest.
  • Key Performance Metrics - Provide key metrics, such as championship titles or race wins.

Dashboard Design and Layout

An outline of what the dashboard should include and how it should be structured was created as a starting point.

Dashboard Structure

The available data had details on races, drivers, and circuits, so it made sense to build something that covered all these areas and allow both a high level view as well as more detailed views when required.

  • Landing Page with Multiple Detailed Pages - The structure was based around a landing page that provided an overview of all the available data that could then allow users to navigate to sub-pages that would provide additional details.

Layout Planning

  • Individual Page Designs - Each page of the dashboard was sketched out to consider what content could be included and how they could be organised on the page. The aim was to use Power BI visuals that made sense for the information being provided and ensure each page had a logical layout that was easy to view.
  • Space Saving Techniques - Due to the amount of information to include it was important to maximise the use of space. This meant the dashboard needed:
    • Collapsible Filter Pane - To allow most of the page to be used for visuals a collapsible filter pane would allow general filters to be included but hidden when not required.
    • Adjustable Visuals - Having visuals that can be toggled from one view to another in the same space on a page would allow for different views using the same page area.

Dark Theme Implementation

  • Dashboard Theme - The overall theme of the dashboard was a dark theme to showcase something slightly different. Power BI has plenty of theme options to pick from and these can be customised where required. In a work situation an organisation would typically have a colour scheme that would dictate what to use.

Data Modelling

The ergast dataset came with its own data model, so that made the initial set up much easier. The data model consists of 14 tables.

Data Import and Setup

The 14 data tables were imported into Power BI and the relationships were set up.

  • Imported Data - The 14 data tables had been downloaded in CSV format from the ergast site. The tables were imported into Power BI with minimal changes. The data formats were generally good so did not need any adjustments. The main thing noted were some fields using text values for null values in columns that were otherwise integers.
  • Relationship Management - Due to the model already being an established and well-modelled set of data, Power BI automatically identified the correct relationships most of the time.

Calculated Columns and Measures

To support the visuals that needed to be created a range of additional calculated columns and measures were created.

  • Calculated Columns - New columns were created where additional values were required to filter or group records to support dashboard metrics. Some examples are:
    • Pole Positions - A new column in the qualifying table to denote the pole position so this could be easily filtered for in visuals.
    • Race Finish Categories - Several columns to help group race finishes into wins, podiums, top tens, finishes and DNFs. These worked in conjunction with calculated measures to make categorised counts easy to create.
    • Flag Images - To include flags in the dashboard a calculated column was created to reference to flag images from flagpedia.net.
  • Calculated Measures - New measures were created to support visuals, often using calculated columns. These included things like measures for race or season finish positions.

Dashboard Development and Refinement

The dashboard content was developed and iterated until the technical content was working and then the final formatting was touched up.

Refining Dashboard Content

The visuals for the dashboard all required some form of technical development and testing. With a range of techniques being used, each one needed to be set up and tested and could then be reused as required across the dashboard.

  • Visual Data - Each visual required a certain value or set of values to be displayed. This required a mix of measures and filters to ensure each visual displayed the correct values.
  • Changing Views with Bookmarks and Buttons - The collapsible filter pane, page navigation, and several visuals with multiple viewing options required the use of bookmarks and buttons. These were combined with layering of visual components that were toggled between being hidden and visible.
  • Filter Settings - There are several different methods for filtering in Power BI. These all need checking to make sure that filters apply to the correct visuals.

Final Formatting

The final step was to make small adjustments to the look and feel of the dashboard to optimise usability. This is where having clear requirements and a good initial design makes it easier to assess if the final product is where it needs to be. Typically users or customers would be consulted through the entire design and development process and their feedback would help confirm what good looks like. In my case, it was just me so the final product is simply an example of what can be created.

  • Formatting - Getting all of the visuals to display with consistent fonts, sizing and spacing. This is important to make sure the dashboard is consistent and users can get used to similar design features.
  • Usability - A review of how a user would navigate around and likely use the dashboard to make sure it makes sense and is easy to use. This is where user feedback would allow a better product to be designed.

Final Result

The Formula One Dashboard consists of 5 pages to allow users to drill into the data as they see fit.

I am using Power BI Desktop and do not have a Power BI license so cannot use Publish to Web to share my dashboard. Instead, some screenshots of the dashboard are provided below to describe how the dashboard works.

Main Page

The landing page for the dashboard provides:

  • Result Tables - Tables for both the driver and constructor champions for each year.
  • Most Championships - The drivers and constructors with the most championships.
  • Circuit Map - A map of all the circuits that have been raced at.
  • Driver Metrics Chart - A chart that can be adjusted to show 5 different driver metrics.

There are also other features that appear on all pages:

  • Page Navigation - There are page navigation buttons in the header bar to allow a user to move from one page to another.
  • Collapsible Filter Pane - The filter pane is on the left side and is collapsed until the arrow button is selected.

Filter Pane

  • When the filter pane is expanded it can house any filters that would be useful across the entire page.
  • Once selected the filters are applied and the filter pane can be collapsed again by clicking on the button or anywhere on the screen.

Main Page - Filtered Visuals

  • Circuit Map - The map can be filtered to regions, or the built in zoom and rectangle select options can be used to zoom in on particular parts of the world.
  • Driver Metrics Chart - This chart has 5 different metrics that can be selected and scrolled through.

Drivers Page

An overview of all drivers that provides:

  • Driver Career Results - A bar chart that can be adjusted to show all races or the cumulative values up to certain placings.
  • Driver Nationalities - A bar chart to display the nationalities of all F1 drivers.
  • Result Tables - Two tables for grand prix results and sprint race results for all drivers. The key metrics are provided and can be used to sort the lists.

Drivers Page - Toggled Visual

  • Driver Career Results - The Driver Career Grand Prix Results chart can be adjusted to show cumulative values so they can be seen in more detail. This allows users to see things like the number of top ten finishes but also the breakdown of wins and podiums within those finishes.

Driver Details

A page where one driver can be selected at a time to view their career in more detail.

  • Driver Drop-down Selector - Drop-down selector for the driver that then returns their nationality, age, and the first and most recent years racing in F1.
  • Race History Table - A table visual that can be toggled between grand prix and sprint race results.
  • Race History Chart - Race history charts by year. The Race History chart can be toggled by finish type with multiple types selectable. The average points and finish per year are shown in a line chart.
  • Additional Driver Stats - Bar charts to display the average finish at each circuit to show a drivers favourite circuits and a detailed breakdown of reasons for not finishing races.

Driver Details - Adjusted Visuals

  • Race History Table - The race history table can be toggled between grand prix and sprint race results.
  • Race History Chart - The race history bar chart can be adjusted to show the finish positions in any combination.

Other Pages

  • Constructors - A page with details on constructors, similar to the Drivers page.
  • Circuits - A page with details on circuits.
